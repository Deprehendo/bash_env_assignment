## What the hooks actually protect

The branch policy is **one-way protection for `main`/`master`**:

| Action | Allowed? |
|--------|----------|
| `git show main:path`, `git diff main`, `git log main` | Yes |
| Read files in the working tree (while on `ai-coding`) | Yes |
| Edit/commit/push on `ai-coding` | Yes |
| `git switch main`, commit on `main`, push to `main` | No |

Your rule and hook both say this explicitly: *“You may read main for reference … but must not check out or edit it.”*

So the guardrails are **“don’t touch main”**, not **“don’t look at main.”**

## Can I copy `main` → `ai-coding`?

Yes, in normal ways:

1. **`git show main:some/file`** — read a file from `main` without checking it out.
2. **Write/StrReplace** — paste that content into the working tree on `ai-coding`.
3. **`git restore --source=main -- path`** — while on `ai-coding`, this is a mutating git command but not on a protected branch, so the hook allows it.

I cannot check out `main` and edit it there, but I can still bring `main`’s content into `ai-coding`. That is often useful (revert a bad change, compare versions, restore a file).

## Is that a “risk”?

It depends what you’re worried about:

**Low risk — accidental damage to `main`**
- I can’t switch to `main`, commit there, merge/rebase there, or push to `main`.
- Your merge workflow stays manual.

**Moderate risk — unwanted changes on `ai-coding`**
- I could overwrite `ai-coding` work with older `main` versions if asked (or if I misunderstand).
- That affects the agent branch only; `main` stays untouched until you merge.

**Separate concern — secrets**
- Branch hooks don’t hide secrets that are committed on `main`.
- `.env`, `.ssh`, etc. are blocked by path-based read/write hooks regardless of branch.
- If credentials live in normal tracked files on `main`, I could read them via `git show main:...` and write them on `ai-coding`. That’s a repo hygiene issue, not a branch-policy gap.

## Practical takeaway

There is **no risk of me silently modifying or pushing `main`**. There **is** intentional ability to **read `main` and apply that content on `ai-coding`**, which is how reference/revert workflows work.

If you want to tighten that further, options would be things like:

- Block `git restore --source=main` / `git checkout main --` patterns in the hook
- Keep secrets out of git entirely (not just on protected paths)
- Use `ai-coding` as a long-lived branch so it diverges cleanly from `main`

For the current setup, **reading and copying from `main` to `ai-coding` is expected behavior**, not a bypass of the protection model.
